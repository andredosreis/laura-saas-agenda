## 🔴 Críticos (6)

### 1. Contraste insuficiente em labels das auth pages glassmorphism
**Ficheiros**: `src/pages/Login.jsx:65`, `Register.jsx:87`, `VerificarEmail.jsx:50`, `ResetPassword.jsx:105`

**Problema**: Labels e helper text com `text-gray-300` em card `bg-white/10` sobre fundo `from-slate-900` cria contraste efectivo de ~3-4:1 (WCAG AA exige 4.5:1 para normal text).

**Impacto**: ~15% dos utilizadores com low vision não conseguem ler labels. Falha auditoria WCAG AA.

**Recomendação**: Trocar `text-gray-300` por `text-white` nos labels, ou aumentar opacidade do card para `bg-white/20` (ou `bg-slate-800/60` para mais opacidade preservando o efeito vidro).

---

### 2. Focus rings invisíveis nos inputs
**Ficheiros**: `Login.jsx:73`, `Register.jsx:95`, `ResetPassword.jsx:173`, `CriarCliente.jsx:88`, `CriarAgendamento.jsx:176`

**Problema**: Inputs têm `focus:ring-2 focus:ring-indigo-500` mas o card `bg-white/10` "engole" o ring visualmente. Keyboard users não vêem focus.

**Impacto**: Keyboard navigation comprometida — fail WCAG 2.4.7 (Focus Visible). Afecta utilizadores que dependem de teclado.

**Recomendação**: Adicionar `focus:ring-offset-2 focus:ring-offset-slate-900` ou aumentar para `focus:ring-4` com `focus:ring-indigo-400` (mais brilhante).

---

### 3. Touch targets abaixo de 44px em botões de acção
**Ficheiros**:
- `src/pages/Clientes.jsx:30-42` — botões "Editar" e "Deletar" com `px-3 py-1` (~28px altura)
- `Login.jsx:173-177`, `Register.jsx:269-276`, `ResetPassword.jsx:176-187` — password toggle (eye icon) com `p-1` (~20px target)

**Problema**: Touch targets reais são ~20-28px. WCAG 2.5.5 e Apple HIG exigem 44×44pt. Em mobile, mis-taps e frustração.

**Impacto**: Críticamente em mobile. Botões de delete <44px são especialmente perigosos (acidente).

**Recomendação**:
- Botões edit/delete: `px-4 py-2.5` mínimo, ou converter em icon buttons com `w-11 h-11` e `aria-label`
- Password toggle: alterar `p-1` para `p-2.5` ou adicionar `min-w-11 min-h-11`

---

### 4. Sem heading hierarchy nem skip link no layout autenticado
**Ficheiro**: `src/App.tsx:42-49` (`ProtectedLayout`)

**Problema**: `<ProtectedLayout>` envolve `<Sidebar>` + `{children}` sem `<main>` semantic, sem skip link, e páginas internas (Dashboard, Clientes) sem `<h1>`.

**Impacto**: Screen reader users sem ponto de referência. Keyboard users sem forma rápida de pular sidebar para conteúdo. Fail WCAG 2.4.1 (Bypass Blocks).

**Recomendação**:
1. Em `App.tsx` envolver `{children}` em `<main id="main-content">`
2. Adicionar antes do Sidebar: `<a href="#main-content" className="sr-only focus:not-sr-only focus:fixed focus:top-2 focus:left-2 focus:z-50 focus:bg-white focus:text-slate-900 focus:px-4 focus:py-2 focus:rounded">Saltar para conteúdo</a>`
3. Cada página principal deve começar com `<h1>` (ex: Dashboard.jsx, Clientes.jsx)

---

### 5. Aria-label em falta em botões icon-only
**Ficheiros**:
- `src/components/Sidebar.jsx:130-140` — toggles de grupo (AGENDAMENTO/FINANÇAS/ADMIN) só com `<ChevronUp/Down>`
- `Sidebar.jsx:197-202` — mobile menu toggle (Menu/X) só com ícone

**Problema**: Screen readers leem apenas "button" para estes controlos. Sem indicação de propósito ou estado expandido/colapsado.

**Impacto**: Navigation crítica indecifrável para screen reader users.

**Recomendação**:
- Group toggles: `aria-label={`${group.label}: ${expandedGroups[group.id] ? 'colapsar' : 'expandir'}`} aria-expanded={expandedGroups[group.id]}`
- Mobile menu: `aria-label={isMobileOpen ? 'Fechar menu de navegação' : 'Abrir menu de navegação'} aria-expanded={isMobileOpen}`

---

### 6. ResetPassword e CriarAgendamento com validação fora do padrão
**Ficheiros**: `ResetPassword.jsx:46-71` (validação manual), `CriarAgendamento.jsx:36-85` (form com state desincronizado)

**Problema**:
- ResetPassword usa `if (password !== confirmPassword)` ao submit, sem feedback enquanto digita. Login/Register usam `react-hook-form` + Zod em `mode: 'onChange'`
- CriarAgendamento mistura `useState` + `Controller` sem validação Zod uniforme; quando cliente muda, pacote select não reseta visualmente

**Impacto**: UX inconsistente — utilizador habituado a feedback inline noutros forms é surpreendido aqui.

**Recomendação**:
- Migrar `ResetPassword` para `useForm` + `resetPasswordSchema` (já existe em `validationSchemas.js:64`)
- `CriarAgendamento`: garantir `key={clienteSelecionado?._id}` no select de pacote para forçar re-render no change

## 🟡 Importantes (13)

### 7. Disabled state em botões só com opacidade
**Ficheiros**: `Login.jsx:217`, `Register.jsx:365`, `VerificarEmail.jsx:81`, `VenderPacote.jsx:532`

**Problema**: `disabled:opacity-50 disabled:cursor-not-allowed disabled:transform-none` mas mantém gradient brilhante. Feedback fraco — utilizador pode tentar double-submit.

**Recomendação**: Adicionar `disabled:from-indigo-400 disabled:to-purple-500` (cores mais pálidas) para indicar estado inactivo claramente.

### 8. Color-only error indication em alguns inputs
**Ficheiros**: `Login.jsx:68-75`, `Register.jsx:90-96`

**Problema**: Inputs com erro só mudam `border-red-500`. Alguns têm `<XCircle>` ícone mas nem todos. Falha WCAG 1.4.1 para utilizadores daltónicos (~8% homens).

**Recomendação**: Adicionar `aria-invalid="true"` quando `errors[field]` existe, e garantir ícone de erro consistente em todos os inputs com erro.

### 9. `min-h-screen` em vez de `min-h-dvh` (iOS Chrome quirk)
**Ficheiros**: Todas as auth pages + Dashboard, Clientes, Configuracoes

**Problema**: `100vh` em iOS Mobile Chrome inclui address bar retrátil → conteúdo bottom pode ser truncado.

**Recomendação**: Substituir `min-h-screen` por `min-h-dvh` (dynamic viewport height) em todos os layouts root.

### 10. Slate-400 em sidebar — contraste borderline
**Ficheiro**: `src/components/Sidebar.jsx:91-96`

**Problema**: `text-slate-400` em estado inactivo dos NavLinks contra fundo escuro tem contraste ~4.5:1 (limite mínimo WCAG AA, sem margem).

**Recomendação**: Trocar para `text-slate-300` (~5.5:1) — preserva look "secondary" mas sobe acima do limite.

### 11. Dashboard sem skeleton screens
**Ficheiro**: `src/pages/Dashboard.jsx:40-52, 91-180`

**Problema**: Dashboard faz 7+ requisições paralelas, mostra spinner genérico durante 2-3s. Skeleton loading existe importado mas não usado.

**Recomendação**: Substituir loader genérico por `<SkeletonKPIGrid />` enquanto `isLoading`.

### 12. Inline error placement inconsistente em forms
**Ficheiros**: comparar `Login.jsx:140-145` (inline limpo) vs `CriarAgendamento.jsx:273` (inline mas dependente de `dirtyFields`)

**Problema**: Algumas páginas mostram erro sempre que `errors[field]` existe; outras só quando `dirtyFields[field]` também é true. Timing ambíguo cria experiência diferente entre forms.

**Recomendação**: Padronizar — sempre que `errors[field]` exists, mostrar `<ErrorMessage>`, sem condicionar a `dirtyFields`.

### 13. Autofill `tel` em falta
**Ficheiros**: `Register.jsx:234`, `CriarCliente.jsx:173`

**Problema**: `<input type="tel" {...register('telefone')} />` sem `autoComplete="tel"`. Browser não preenche automaticamente.

**Recomendação**: Adicionar `autoComplete="tel"` em todos os inputs de telefone. Para email, `autoComplete="email"`. Para nome `autoComplete="name"`.

### 14. Password strength sem aria-live
**Ficheiros**: `Register.jsx:281-302`, `ResetPassword.jsx:225-252`

**Problema**: Strength bars são puramente visuais. Screen readers não anunciam progresso da força.

**Recomendação**: Wrap do indicator com `aria-live="polite" aria-label={`Força da senha: ${passwordStrength.label}`}`.

### 15. CriarAgendamento mobile cramped
**Ficheiro**: `CriarAgendamento.jsx:218`

**Problema**: `grid grid-cols-2 gap-2` para toggle de tipo de agendamento ficam comprimidos em <480px.

**Recomendação**: Usar `grid-cols-1 sm:grid-cols-2` em mobile, com botões full-width.

### 16. Sidebar mobile — scroll de fundo activo com drawer aberto
**Ficheiro**: `Sidebar.jsx:194-235`

**Problema**: Drawer mobile tem z-50, overlay z-40, mas body continua scrollable. Em mobile, scroll de fundo "vaza" através do drawer.

**Recomendação**: Quando `isMobileOpen` for true, adicionar `useEffect` que aplica `document.body.style.overflow = 'hidden'` (e remove no cleanup).

### 17. Mobile nav sem `aria-current="page"`
**Ficheiro**: `Sidebar.jsx:91-96`

**Problema**: `NavLink` activo tem visual mas não estado semântico. Após fechar drawer mobile, utilizador screen-reader não sabe a current page.

**Recomendação**: Adicionar `aria-current="page"` quando `isActive` no NavLink (alguns implementations do react-router já fazem isto automaticamente — verificar se está activo).

### 18. Clientes sem empty state com CTA
**Ficheiro**: `src/pages/Clientes.jsx:120-142`

**Problema**: Lista vazia (novo tenant) mostra apenas search box em branco, sem mensagem útil.

**Recomendação**: Adicionar bloco condicional:
```
{!isLoading && clientesFiltrados.length === 0 && (
  <div>Ainda não tens clientes. <Link to="/clientes/novo">Criar o primeiro</Link></div>
)}
```

### 19. `window.confirm` em delete cliente
**Ficheiro**: `src/pages/Clientes.jsx:106-114`

**Problema**: Confirmação de delete via `window.confirm()` — bloqueador, sem context (qual cliente vai ser apagado), inconsistente com resto da app que usa toasts/modais.

**Recomendação**: Implementar modal customizado com:
- Nome do cliente referenciado
- Botão "Cancelar" (secundário)
- Botão "Eliminar" em vermelho destacado
- `role="alertdialog"` com `aria-labelledby` e `aria-describedby`

## 🟢 Polimento (5)

### 20. Animações framer-motion sem `useReducedMotion`
**Ficheiro**: `Sidebar.jsx:146-150`

**Recomendação**: Wrap motion.div animations com hook `useReducedMotion()` (já disponível em framer-motion) — duração 0 quando user prefere.

### 21. Spinner sem `motion-safe:animate-spin`
**Ficheiros**: vários

**Recomendação**: Trocar `animate-spin` por `motion-safe:animate-spin` em loading spinners.

### 22. Cores hardcoded em `InstallPrompt.tsx`
**Ficheiro**: `src/components/InstallPrompt.tsx:76-77`

**Recomendação**: Trocar gradients fora do design system pelo padrão `from-indigo-500 to-purple-600`.

### 23. Recharts sem legend visible em mobile
**Ficheiro**: `src/pages/Financeiro.jsx:19-21`

**Recomendação**: `<Legend layout="vertical" verticalAlign="bottom" />` para mobile.

### 24. Zod error messages mostram só primeiro erro
**Ficheiro**: `validationSchemas.js:45-49`

**Recomendação**: (opcional) Mostrar todos os erros de password simultaneamente em vez de só o primeiro encontrado.
