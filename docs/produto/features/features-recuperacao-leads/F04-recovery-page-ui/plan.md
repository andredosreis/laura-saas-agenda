# F04 — Recovery Page UI — Plan

**Spec:** `./spec.md` · **Complexity:** medium · **Phases:** 3

## Prerequisites
- F02 (`GET /leads/recuperacao`) and F03 (`GET /leads/recuperacao/export.csv`) implemented and reachable — this feature has no backend work of its own.
- Verify the F03 route sets `Access-Control-Expose-Headers: X-Export-Truncated, Content-Disposition` — without it the truncation toast (spec §5.6, §6) silently never fires even though the file downloads correctly. Flag to the F03 owner if missing; do not attempt to work around it client-side.
- Patterns confirmed: `pages/LeadsKanban.tsx` (page conventions — `ThemeContext.isDarkMode`, `ErrorBoundary`, `react-toastify`), `services/leadsService.ts` (typed `api.js` wrapper style), `types/lead.ts` (existing `LeadOrigem`/`LeadStatus`/`LEAD_STAGES` to reuse, not redeclare), `components/Sidebar.jsx` (`menuGroupsAll` + `hasPermissao()` gating), `App.tsx` (lazy route registration), `pages/Atendimentos.jsx` (prev/next pagination control), `pages/Transacoes.jsx` (stat-tile grid responsive convention + existing, non-reusable client-side CSV pattern to *not* copy — no BOM/sanitization there).
- Frontend dev server running per `CLAUDE.md` → Environment (`cd laura-saas-frontend && npm run dev`).

## Phase 1 — Types & data-access layer
1. **Types** — Create `laura-saas-frontend/src/types/recuperacaoLeads.ts` per spec §3: `LEAD_MOTIVO_CODES`/`RECUPERACAO_MOTIVO_LABELS` (mirroring PRD §4.1's 8 codes), `RecuperacaoPorMotivo`, `RecuperacaoResumo`, the discriminated `RecuperacaoLeadRow` (`RecuperacaoLeadPerdido` | `RecuperacaoLeadEsfriado`), `RecuperacaoLeadsResponse`, `RecuperacaoFiltros`. Import `LeadOrigem`/`LeadStatus` from the existing `./lead` rather than redeclaring them.
2. **Service** — Create `laura-saas-frontend/src/services/recuperacaoLeadsService.ts`: `getRecuperacao(filtros): Promise<RecuperacaoLeadsResponse>` (`api.get('/leads/recuperacao', { params })`), `buildQueryParams(filtros)` shared helper that drops empty/undefined keys and strips `page`/`limit` for export calls, and `downloadExportCsv(filtros): Promise<{ truncated: boolean }>` implementing spec §5.6's blob-download flow (axios `responseType: 'blob'` → read `X-Export-Truncated` header → build `Blob` + `URL.createObjectURL` → temporary `<a download>` click, filename parsed from `Content-Disposition` with a Luxon `Europe/Lisbon`-dated fallback → `URL.revokeObjectURL`).

## Phase 2 — Presentational components & page
3. **Summary strip** — Create `laura-saas-frontend/src/components/leads/RecuperacaoSummaryStrip.tsx`: 7 glass-card tiles from a `RecuperacaoResumo` prop, fixed order per spec §5.2, `taxaConversao` formatted as a percentage, responsive `grid-cols-2 md:grid-cols-4 lg:grid-cols-7`.
4. **Reason breakdown** — Create `laura-saas-frontend/src/components/leads/RecuperacaoMotivoBreakdown.tsx`: horizontal bar per `porMotivo` entry (width relative to the max total in the array), empty-state message when the array is empty. Plain divs, indigo/purple fill — no chart library.
5. **Filters bar** — Create `laura-saas-frontend/src/components/leads/RecuperacaoFiltrosBar.tsx`: controlled inputs for `de`/`ate` (native date inputs), `grupo` (segmented control/select), `motivoCodigo` (select from `RECUPERACAO_MOTIVO_LABELS`), `origem` (select from the existing `LEAD_ORIGEM`); emits the full next `RecuperacaoFiltros` object on any change (parent resets `page` to 1).
6. **Lead row** — Create `laura-saas-frontend/src/components/leads/RecuperacaoLeadRow.tsx`: single row rendering nome/telefone/etapaParada/motivo(+expandable nota)/diasParado, an `origem` tag, a `jaContactado` badge when true, and an empty trailing actions region reserved for F05. CSS-driven table row on `md:` and up, stacked card below it (spec §5.5) — avoid a second component just for the mobile variant.
7. **Page** — Create `laura-saas-frontend/src/pages/RecuperacaoLeads.tsx`: owns `filtros`/`page` state (`useState`), fetch via `useCallback` + `useEffect` on `[filtros, page]` (pattern from `LeadsKanban.tsx`), `isLoading`/`error` state, permission-denied branch on a mocked/received `403`, empty-state branch on `data.leads.length === 0`, wraps the tree in `ErrorBoundary`, wires the "Exportar CSV" button to `recuperacaoLeadsService.downloadExportCsv` with the truncation `toast.warning(...)`, renders `RecuperacaoSummaryStrip` → `RecuperacaoMotivoBreakdown` → `RecuperacaoFiltrosBar` → list of `RecuperacaoLeadRow` → pagination controls (prev/next + "Página X de Y", per `Atendimentos.jsx`).

## Phase 3 — Routing, navigation, verification
8. **Route** — In `laura-saas-frontend/src/App.tsx`, add `const RecuperacaoLeads = lazy(() => import('./pages/RecuperacaoLeads'));` next to the other `/leads/*` lazy imports, and `<Route path="/leads/recuperacao" element={<ProtectedLayout><RecuperacaoLeads /></ProtectedLayout>} />` grouped with the existing `/leads`, `/leads/kanban`, `/leads/:id` routes — no extra `allowedRoles`/`requiredPlans` prop (matches its siblings).
9. **Sidebar entry** — In `laura-saas-frontend/src/components/Sidebar.jsx`, add one item to the `crm` group's `items` array: `{ to: "/leads/recuperacao", text: "Recuperação", icon: <an unused lucide icon, e.g. RotateCcw>, perm: 'verLeads' }`. Import the icon from `lucide-react` alongside the existing imports.
10. **Gates** — Run `npm run lint` and `npm run build` (TypeScript check + Vite build) inside `laura-saas-frontend/` until clean; then verify the contract.md criteria (Playwright-drivable checks per spec §7) before handing off to `/implement-evaluate`.
