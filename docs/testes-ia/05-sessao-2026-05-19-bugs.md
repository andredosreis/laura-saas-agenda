## Bugs encontrados (ordem do backlog)

### #1 — F12 router intercepta mensagens com "ok/sim/perfeito/..."

**Severidade:** 🔴 Alta — 2 ocorrências em conversas diferentes.

**Sintoma:** Lead a meio de uma conversa com a IA diz *"ok obrigado pela sua atenção"* ou *"ok agradeço, mas nao vou deixar para a proxima"*. O classifier do F12 vê o "ok" inicial e classifica como `confirmation_yes`. O router então despacha para `NO_PENDING_APPOINTMENT_REPLY` em vez de continuar com `IA_LEAD`. O lead recebe resposta robótica: *"Recebi a sua mensagem, mas não encontrei nenhum agendamento pendente"*.

**Causa raiz:** `src/modules/messaging/routing/messageRouter.js:138-150` — o passo 3 short-circuita para o handler de confirmação SEM verificar se há Lead IA activo. PALAVRAS_SIM no classifier inclui "ok", "perfeito", "claro" — palavras que aparecem naturalmente em conversa.

**Fix aplicado:** mudar a condição de `if (isConfirmation)` para `if (isConfirmation && persistedState.hasPendingAppointment)`. Quando não há appointment pendente, cai nos passos 4-7 onde Lead activo → `IA_LEAD`.

**Verificação:** `tests/message-router.test.js` actualizado com 3 novos casos (SIM sem appointment + sem Lead → IA_LEAD; NÃO sem appointment + sem Lead → IA_LEAD; SIM sem appointment + Lead activo → IA_LEAD). 17/17 testes passam.

### #2 — Extractor classifica distância como `desistir` em vez de `hesitacao`

**Severidade:** 🟡 Médio.

**Sintoma:** Lead diz *"moro em Lisboa, para mim nao da"* — extractor classifica `intent=desistir, score_delta=-30`. O agent ainda assim oferece flexibilidade (regra do prompt), mas o lead é prematuramente movido para perto de `perdido` no primeiro sinal de distância.

**Causa raiz:** prompt do extractor (`lead_extractor.py`) trata qualquer recusa como `desistir`. Não distingue *primeiro sinal* (oportunidade de superação) de *recusa repetida* (desistência real).

**Fix aplicado:** regra dura no extractor — *"distância é `hesitacao` na primeira menção, só vira `desistir` se o lead REPETIR a recusa ou disser claramente que não vai"*. 3 novos fixtures em `distancia_hesitacao.json` cobrem os casos.

### #3 — `Lead.status` não vai para `agendado` por causa do regex BOOKING

**Severidade:** 🟡 Médio.

**Sintoma:** Agent cria agendamento via `create_appointment` tool com sucesso (Mongo tem Agendamento), mas `Lead.status` fica em `em_conversa` em vez de `agendado`. Bug visível na conversa Cintia.

**Causa raiz:** `_BOOKING_REGEX` em `ia-service/.../lead_orchestrator.py` usava `[^.!?]*?` no lookahead. A resposta *"Está marcado, Cintia! 🎉 *Segunda-feira, dia 25, às 09:00.*"* contém `!` após o nome, terminando o lazy match antes de chegar a `09:00`. Resultado: regex falha → `_maybe_auto_book` não move stage.

**Fix aplicado:** mudar para `[^.?]{0,120}?` — só `.` e `?` terminam o look-ahead (são separadores de frases), `!` é tolerado. Cap de 120 chars para não puxar HH:MM de muito longe.

**Verificação:** 6/6 test cases passam (incluindo a frase original da Cintia).

### #4 — Agent inventa distância sem tool de mapas

**Severidade:** 🟢 Baixo localmente, **🟡 Médio em casos extremos**.

**Sintoma:** Cintia perguntou distância de Porto de Mós a Leiria. Agent respondeu *"15 km, 20-25 min de carro"* — sem ter tool de cálculo de distância. Próximo do real (~25 km) mas é informação fabricada.

**Causa raiz:** prompt não tem regra explícita para "lead pergunta distância". Agent improvisa.

**Fix aplicado:** novo guard-rail "📏 Distância / localização" no `system_lead_agent.md` proibindo inventar km/min e obrigando a dar apenas morada + link Maps.

### #5 — Agent não gere expectativa de "resultado/sessões"

**Severidade:** 🟡 Médio — má execução, não viola regra dura.

**Sintoma:** Luana perguntou *"em quantas sessões eu fico lisinha?"*. Agent respondeu com a fórmula de preço (40 €) que ela não pediu. Não admitiu que não pode prometer, não recusou explicitamente, apenas redireccionou.

**Causa raiz:** prompt tem regra "nunca prometer resultados" mas falta **frase modelo concreta** para o caso. Agent improvisa → cai na única fórmula que conhece bem (preço).

**Fix aplicado:** novo guard-rail "⏱ Resultado garantido / número de sessões" no `system_lead_agent.md` com frase modelo em 3 partes: reconhecer expectativa + recusar com PORQUÊ + redirect.

### #6 — Agent recomenda tratamento para condição médica sem disclaimer

**Severidade:** 🔴 **CRÍTICA — risco legal e de segurança**.

**Sintoma:** Julia disse *"eu tenho diabetes e tem qual o massagem ajudaria-me mais"*. Agent respondeu *"Para o seu caso, a massagem que pode ser mais benéfica é a massagem terapêutica, que ajuda a melhorar a circulação e a aliviar tensões"* — recomendação clínica directa, sem mencionar contraindicações reais da diabetes (neuropatia, risco de feridas), sem pedir autorização médica.

**Causa raiz:** prompt tem regra "nunca inventar contraindicações" mas NÃO tem regra simétrica para o caso de *lead reportar condição médica*. Agent trata como pergunta normal de produto.

**Fix aplicado:** novo guard-rail crítico "🩺 Condições médicas reportadas" no `system_lead_agent.md` com:
- Lista explícita de condições gatilho (diabetes, hipertensão, gravidez, pós-op, trombose, cancro, ferida aberta, autoimune, imunossupressores)
- Frase modelo obrigatória: empatia + reconhecimento de avaliação personalizada + **pedido de autorização médica**
- 4 proibições explícitas (não recomendar tratamento, não prometer benefício, não dizer que é seguro, não diagnosticar)

3 novos fixtures em `condicao_medica.json` (diabetes, gravidez, pós-lipo) garantem cobertura no LangSmith.

### #7 — Extractor não destaca condição médica em `observacoes`

**Severidade:** 🟡 Médio — informação visível à Laura ficava enterrada no `motivoInteresse`.

**Sintoma:** Para a Julia, o extractor pôs "diabetes" só no `motivo` ("massagem que ajude com diabetes"). A Laura abrindo o Kanban vê motivo genérico — não vê *"⚠ condição médica reportada"* destacado.

**Fix aplicado:** schema `LeadIntel.observacoes` descrição estendida — quando lead reporta condição médica, extractor preenche `observacoes` com prefixo `⚠ Condição médica: <descrição>`. Convenção testada nos 3 fixtures de `condicao_medica.json` — todos os 3 passaram a captura correcta.

## Decisões de arquitectura registadas

1. **Prompt-as-design** sustentado — todos os 7 fixes foram resolvidos
   por afinamento de prompts e 1 mudança cirúrgica de routing. Nenhum
   precisou de mudança de arquitectura (LangGraph etc.).

2. **F12 router precedence** — Confirmation routing só intercepta
   quando há `hasPendingAppointment=true`. Caso contrário, IA_LEAD
   tem precedência. Esta mudança simplifica o decision tree e elimina
   o falso positivo de "ok/sim/perfeito" dentro de conversação.

3. **Eval-driven prompt evolution** — cada fix de prompt foi
   acompanhado de fixture novo no LangSmith para garantir que não
   regride no futuro. Os guard-rails críticos (#6 médico) ganharam
   3 examples dedicados.

4. **Observabilidade da condição clínica** — a convenção
   `⚠ Condição médica: ...` no `Lead.observacoes` é o sinal canónico
   que a Laura usa no Kanban para identificar leads que requerem
   cuidado clínico extra.
