## Bugs detectados antes dos fixes

### Bug original 1 — `/api/internal/leads` 404

A branch F12 só tinha o módulo Leads parcial. Sem `leadInternalRoutes.js`,
o Python falhava silenciosamente a criar Leads (response 200 com
`status:error, action_taken:lead_create_failed`).

**Causa:** F12 PR não incluía o módulo completo, só os models cherry-picked.
**Fix:** merge local de `feat/leads-phase4a-knowledge-base` na branch F12.
**Status:** ✅ Resolvido.

### Bug original 2 — Agent perdia o nome entre turns

Conversa com "Maria":
- Turn 2: "Maria" → "Olá Maria! 😊" (capturou)
- Turn 7+: "estou com dores" → agent volta a perguntar "qual o seu nome?"

**Causa:** o agent era stateless por invocation. Não recebia `Lead.nome`
como variável estruturada no system prompt — só via via conversation
history, que o LLM ignorava quando o user mudava de tópico.

**Fix aplicado:** injectar `{{lead_nome}}`, `{{lead_motivo}}`, `{{lead_urgencia}}`,
`{{lead_score}}` no system prompt. Adicionada regra absoluta no topo:
*"Se Nome NÃO é (ainda não recolhido), PROIBIDO pedir o nome de novo."*

**Status:** ✅ Resolvido para casos onde Lead.nome é persistido no DB.

### Bug original 3 — Extractor classificou "as 9 da?" como `primeira_msg`

Conversa "casamento próxima semana":
- Turn 5: "as 9 da?" (lead a escolher slot) → extractor classificou
  `intent=primeira_msg` → agent reset: *"Olá! 😊 Em que posso ajudar?"*

**Causa:** `EXTRACTOR_SYSTEM_PROMPT` mandava focar **apenas na última
mensagem**, ignorando contexto. LLM via `"as 9 da?"` isoladamente e
classificava como saudação inicial.

**Fix aplicado:** reescrever extractor prompt para olhar history antes
de classificar. Lista explícita de exemplos (mensagens curtas com `?`
após proposta de slots → `escolher_slot`, não `primeira_msg`).

**Status:** ✅ Resolvido para os casos testados.

## Conclusão arquitectural

Tentar resolver tudo via prompt cada vez mais defensivo **não está a
escalar**. Cada cenário novo expõe um caso que o prompt não cobriu.

**Recomendação forte**: migrar `lead_agent.py` para **LangGraph** com
state machine explícita (nós: `collect_name`, `discovery`, `propose_slot`,
`confirm_booking`, `handle_objection`). Cada nó conhece o seu objectivo
e o que falta capturar. As transições são determinísticas, não dependem
de o LLM interpretar correctamente uma regra de prompt entre dezenas
de outras.

Ver `04-proximos-passos.md` para o plano de migração proposto.
